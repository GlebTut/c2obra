The[a][b] Design[c] Question for switch[d][e]

Here are all possible ways to implement switch coverage instrumentation, from simplest to most complete:

Option 1 — Case Hit Only (Simplest)

Track only whether each case label was reached. No wrapping of the switch expression.

c

// Original

switch (x) {

    case 1:  do_a(); break;

    case 2:  do_b(); break;

    default: do_c(); break;

}

// Instrumented

switch (x) {

    case 1:  __coverage_hit(1); do_a(); break;

    case 2:  __coverage_hit(2); do_b(); break;

    default: __coverage_hit(3); do_c(); break;

}

branch_map.json:

json

{ "id": 1, "line": 6, "type": "switch_case", "label": "case 1" },

{ "id": 2, "line": 7, "type": "switch_case", "label": "case 2" },

{ "id": 3, "line": 8, "type": "switch_case", "label": "default" }

Runtime: just __coverage_hit(id) — increments branch_counters[id*2]

Option 3 — Model Each Case as a Binary Branch

Treat each case exactly like an if — true if that case ran, false if it didn't.

    case 1:

        cover(x == 1, 1);   // true if case 1 ran

        do_a(); break;

    case 2:

        cover(x == 2, 2);   // true if case 2 ran

        do_b(); break;

    default:

        cover(x != 1 && x != 2, 3);  // true if default ran

How TestCov Handles switch — Research Findings

Key Discovery from the TestCov Paper (Beyer & Lemberger, ASE 2019)

TestCov — the official coverage measurement tool used in Test-COMP — does not use GCC's built-in branch tracking for switch. Instead, it uses a label injection approach:​

"TESTCOV adds program labels BRANCH_i at the beginning of each program branch of a program and uses the line-coverage measurement of LCOV to check which of the added program labels are covered."[j]

In code form, TestCov instruments a switch like this:

c

// Original

switch (x) {

    case 1: do_a(); break;

    case 2: do_b(); break;

}

// TestCov instrumentation

switch (x) {

    case 1: BRANCH_1:; do_a(); break;   ← label injection

    case 2: BRANCH_2:; do_b(); break;   ← label injection

}

It then uses lcov line coverage to check which BRANCH_i labels were reached. This is exactly Option 1 — one entry per case, no wrapping of the switch expression.

Why TestCov Does It This Way

The paper explicitly explains the reason:​

"LCOV claims to store branch coverage, but considers each condition of a short-circuit boolean operation as a separate branch... LCOV reports a branch coverage of only 25% instead of the expected 50%. To circumvent this, TESTCOV adds program labels."

So TestCov avoids wrapping the switch expression entirely — it only injects labels at each case. This validates Option 1 as the industry standard for Test-COMP tools.

Proposed Design for Our Tool (Matching TestCov)

c

// Original

switch (x) {

    case 1:  do_a(); break;

    case 2:  do_b(); break;

    default: do_c(); break;

}

// Instrumented (matching TestCov approach)

switch (x) {

    case 1:  __coverage_hit(1); do_a(); break;

    case 2:  __coverage_hit(2); do_b(); break;

    default: __coverage_hit(3); do_c(); break;

}

__coverage_hit(id) is our equivalent of TestCov's BRANCH_i: label — it records that this case was reached.

Let's think this through carefully before writing a single line.

The 5 Problems We Need to Solve

Problem 1 — What to inject for default

default has no condition. Two options:

c

else if (cover(1, 3)) ;     // always true — records default branch was possible

// or

else ;                       // no cover() — default is implicit

Problem 2 — What if there is no default

c

switch (x) {

    case 1: do_a(); break;

    case 2: do_b(); break;

    // no default!

}

We still need to record the "no case matched" branch. The else at the end covers this:

c

if (cover(x == 1, 1)) ;

else if (cover(x == 2, 2)) ;

else if (cover(1, 3)) ;     // implicit default — x didn't match any case

switch (x) { ... }

Problem 3 — Fall-through cases (no value)

c

case 1:       // no value — falls through to case 2

case 2: do_a(); break;

Here case 1 has no condition of its own. We need to detect this in the AST and skip injecting a cover() for it, OR inject cover(x == 1, id) and treat it as a separate branch that happens to share behaviour with case 2.

Decision: inject cover(x == 1, id) for every case label that has a value, even if it falls through.

Problem 4 — Extracting case values from the AST

For case 1:, tree-sitter gives us the node case_statement with a value child containing 1. We need to extract that value as a string to build x == 1.

For the switch expression switch (x), we need to extract x as a string.

tree-sitter node structure:

text

switch_statement

├── "switch"

├── parenthesized_expression  ← this is "(x)" — extract inner as "x"

└── compound_statement

    ├── case_statement

    │   ├── "case"

    │   ├── [value node]      ← this is "1"

    │   ├── ":"

    │   └── ...body...

    └── default_statement

        ├── "default"

        ├── ":"

        └── ...body...

Problem 5 — Where exactly to inject the if/else if block

We need to insert text immediately before the switch keyword — at switch_node.start_byte. The SourceRewriter already supports this since inserting at position X with replace(X, X, text) is a zero-width insertion.

Step 1 — Extract switch metadata from AST

Write get_switch_cases(switch_node, source_code) that returns:

        {"default": True, "line": 8}   # or None if no default

    ],

    "has_default": True

}

Step 2 — Generate the synthetic if/else if block

Write build_cover_chain(switch_meta, base_id) that returns:

c

else if (cover(x == 2, 2)) ;

else if (cover(1, 3)) ;

as a Python string, ready to inject.

Step 3 — Inject using SourceRewriter

python

insert_pos = switch_node.start_byte

rewriter.replace(insert_pos, insert_pos, cover_chain + "\n")

Step 4 — Update write_branch_map()

Each case and the default get their own branch map entry:

json

{ "id": 1, "line": 6, "type": "switch_case", "label": "case 1" },

{ "id": 2, "line": 7, "type": "switch_case", "label": "case 2" },

{ "id": 3, "line": 8, "type": "switch_default", "label": "default" }

Step 5 — Update ID allocation

Current system assigns 1 ID per branch. A switch with N cases + default now needs N+1 IDs. We need a pre-pass to count total IDs before instrumenting:

python

def count_ids_needed(branches):

    total = 0

    for b in branches:

        if b['type'] == 'switch_statement':

            total += count_cases(b['node'])  # N cases + default

        else:

            total += 1

    return total
